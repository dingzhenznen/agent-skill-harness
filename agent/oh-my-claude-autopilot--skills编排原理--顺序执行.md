  autopilot skill 的编排机制

  根据 skills/autopilot/SKILL.md，它实现了6 个阶段的完整编排流程：

  ┌──────┬──────┬──────────┬──────────────┐

  ┌─────────┬────────────┬──────────────────────────────────┬───────────────────────────────────────────────┐
  │  阶段   │    名称    │             工作内容             │                 使用的 Agent                  │
  ├─────────┼────────────┼──────────────────────────────────┼───────────────────────────────────────────────┤
  │ Phase 0 │ Expansion  │ 将用户的想法扩展为详细的需求规格 │ Analyst (Opus) + Architect (Opus)             │
  ├─────────┼────────────┼──────────────────────────────────┼───────────────────────────────────────────────┤
  │ Phase 1 │ Planning   │ 创建实现计划                     │ Architect (Opus) + Critic (Opus)              │
  ├─────────┼────────────┼──────────────────────────────────┼───────────────────────────────────────────────┤
  │ Phase 2 │ Execution  │ 并行执行实现                     │ Executor (Haiku/Sonnet/Opus)                  │
  ├─────────┼────────────┼──────────────────────────────────┼───────────────────────────────────────────────┤
  │ Phase 3 │ QA         │ 循环测试修复                     │ UltraQA 模式                                  │
  ├─────────┼────────────┼──────────────────────────────────┼───────────────────────────────────────────────┤
  │ Phase 4 │ Validation │ 多视角代码审查                   │ Architect + Security-reviewer + Code-reviewer │
  ├─────────┼────────────┼──────────────────────────────────┼───────────────────────────────────────────────┤
  │ Phase 5 │ Cleanup    │ 清理状态文件                     │ -                                             │
  └─────────┴────────────┴──────────────────────────────────┴───────────────────────────────────────────────┘

  关键编排特性

  1. 顺序执行：每个阶段必须完成后才能进入下一个阶段
  2. 并行优化：在 Phase 2（Execution）中会使用 ralph + ultrawork 来并行处理独立任务
  3. 质量门控：
    - QA 阶段最多循环 5 次，同一错误出现 3 次会停止并报告
    - Validation 阶段需要所有 reviewer 通过才算完成
  4. 状态管理：通过 .omc/autopilot-state.json 等文件保存进度，支持中断后恢复
  5. 快捷路径：如果检测到已存在的 ralplan 共识计划或 deep-interview 规格文档，会跳过相应阶段直接执行


    用户输入 "/autopilot"
      ↓
  主 Agent (Claude) 识别到关键词 "autopilot"
      ↓
  调用 Skill 工具 → 加载 `autopilot/SKILL.md`
      ↓
  Skill 定义编排流程（6个Phase + 各Phase使用的子Agent）
      ↓
  主 Agent 按照 Skill 的指引，调用相应的子 Agent 完成任务

  主 Agent vs Skill 的职责分工

  ┌──────────────────────┬────────────────────────────────────────────────────────────────────────┐
  │         层级         │                                  职责                                  │
  ├──────────────────────┼────────────────────────────────────────────────────────────────────────┤
  │ 主 Agent (Claude)    │ 识别用户意图、加载对应的 Skill、按照 Skill 指引执行工具调用            │
  ├──────────────────────┼────────────────────────────────────────────────────────────────────────┤
  │ Skill (如 autopilot) │ 定义完整的编排逻辑：阶段划分、Agent 选择、执行顺序、质量门控、状态管理 │
  ├──────────────────────┼────────────────────────────────────────────────────────────────────────┤
  │ 子 Agent             │ 执行具体任务（分析、规划、编码、审查等）                               │
  └──────────────────────┴────────────────────────────────────────────────────────────────────────┘

  关键点

  - 主 Agent 不内置编排逻辑，而是委托给 Skill
  - Skill 相当于**"剧本"**，告诉主 Agent 在什么阶段调用什么 Agent
  - 同一个 Skill 可以被多次调用，保持行为一致

  这正是 OMC 设计的核心：Skill 负责编排，主 Agent 负责执行编排
