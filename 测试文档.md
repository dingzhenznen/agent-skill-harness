需要，通常 最好先给，否则 AI 做自动化测试时很容易出现这几类问题：

  - 不知道系统“正确行为”是什么，只能按代码表面去猜
  - 只会补“存在性测试”或“happy path”
  - 漏掉边界条件、异常路径、权限和状态流转
  - 写出和业务目标不一致的测试

  但这里的“测试文档”不一定要很重。关键不是文档形式，而是要让 AI 拿到足够明确的测试依据。

  最少应该给 AI 这些信息：

  1. 需求/spec
     告诉它这个功能应该怎么工作，什么算正确。
  2. 验收标准
     最好是可验证的规则，比如：

  - 库存不足时下单失败
  - 重复扣减请求不能重复扣库存
  - 已下架商品不可购买

  3. 关键业务流程
     比如成功路径、失败路径、回滚路径。
  4. 边界与异常
     比如空值、非法状态、并发、幂等、权限不足。
  5. 测试范围
     告诉 AI 这次要写什么测试：

  - 单元测试
  - 集成测试
  - API 测试
  - E2E 测试

  6. 已有测试约定
     比如：

  - 用什么框架
  - 测试目录结构
  - mock 策略
  - fixture 怎么组织
  - 命名规范

  如果没有正式测试文档，也至少要给 AI 一个“测试说明”小节。对很多项目来说，这就够了。

  一个很实用的最小模板是：

  ## Test Guidance

  ### Scope
  - 商品创建
  - SKU 编辑
  - 库存扣减
  - 库存回滚

  ### Happy Path
  - 创建商品成功
  - 支付成功后库存扣减成功

  ### Edge Cases
  - 缺少必填字段
  - 库存不足
  - 重复请求幂等
  - 商品已下架

  ### Non-Goals
  - 不测前端页面
  - 不测真实第三方支付

  ### Test Type
  - 单元测试覆盖领域规则
  - 集成测试覆盖库存扣减与回滚

  结论很简单：

  - 简单场景：不一定要正式测试文档，但至少要有测试说明
  - 涉及业务规则的系统：最好先给 AI 测试文档或测试策略
  - 库存、订单、支付这类强约束场景：建议明确写测试文档，否则 AI 很容易漏关键用例
