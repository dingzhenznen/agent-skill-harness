UI文档应该根据prd文档来设计还是根据spec文档来设计

**UI 文档应该主要依据 Spec 来设计**，同时以 PRD 为上下文参考。更准确地说：

- **PRD** 提供“为什么做”和“做什么功能”（用户目标、业务价值、功能范围）。
- **Spec** 提供“具体怎么做”（输入校验、错误提示、状态变化、流程跳转、字段规则等）。

### 为什么主要依据 Spec？

1. **Spec 已经给出了 UI 需要承载的交互逻辑**  
   例如：Spec 写明“手机号输入框失焦时校验格式，若错误则显示红色文案”。UI 设计师据此设计该输入框的布局、报错样式、以及正常/错误状态的视觉差异。

2. **PRD 中的需求描述往往不够精确**  
   PRD 可能只说“用户可以登录”，不明确“密码错误 3 次后是否锁定”。如果 UI 只参考 PRD，可能会漏掉异常状态的页面设计（如锁定提示页）。

3. **Spec 是开发、测试的直接输入**  
   UI 文档与 Spec 对齐，能保证界面表现与功能逻辑一致，避免后期返工。

### 实际流程建议

| 步骤 | 输入 | 输出 |
|------|------|------|
| 1. 产品经理编写 PRD | 用户调研、业务需求 | PRD（功能清单、用户故事） |
| 2. 技术/产品细化 Spec | PRD + 技术约束 | Spec（交互行为、API 定义、规则） |
| 3. UI 设计师同时查阅 PRD 和 Spec，**以 Spec 为准** | PRD（理解目标）+ Spec（逻辑规则） | UI 设计稿（含正常/异常/空状态等） |

**举例：**  
- PRD 说“用户注册”。  
- Spec 说“注册需填写用户名(6-20字符)、密码(至少8位含数字)、两次密码需一致；提交前校验；提交失败时保留已填内容并显示具体错误”。  
- UI 据此设计表单字段、实时校验提示、失败弹窗等——这些细节 PRD 并没有。

### 结论

- **主要依据：Spec**（因为它定义了 UI 必须呈现的所有行为规则和边界条件）。  
- **辅助参考：PRD**（帮助理解用户心理模型和业务优先级，从而做出合理的视觉与信息布局）。  
- 二者不是对立关系，而是上下游：PRD → Spec → UI 设计稿。



那技术设计文档应该根据prd文档还是根据spec文档来设计



**技术设计文档应该主要依据 Spec，并参考 PRD 中的非功能性需求与业务背景。**

### 详细解释

- **Spec** 给出了具体的功能行为：输入输出、边界条件、异常处理、业务流程、API 定义、数据规则等。技术设计需要逐一给出实现方案——例如 Spec 说“密码连续错5次锁定30分钟”，技术设计就要设计锁定标志的存储方式、计时机制、解锁逻辑等。
- **PRD** 提供用户目标、业务价值和**非功能性需求**（如并发量、响应时间、安全性、数据保留周期），这些会影响技术选型和架构设计。但 PRD 不包含实现所需的细节，无法单独作为技术输入。

因此，**最佳实践是**：

| 文档 | 对技术设计的贡献 |
|------|----------------|
| **Spec** | 功能细节、逻辑规则、API 契约、数据字段 → **必读，主要依据** |
| **PRD** | 业务背景、性能/安全/合规要求、用户规模、部署环境 → **参考，用于非功能设计** |

### 实际协作流程

1. 产品输出 PRD（业务视角）
2. 技术负责人/系统工程师基于 PRD 写出 Spec（技术-产品视角，细化规则）
3. 开发工程师基于 Spec 撰写技术设计文档（实现视角）

### 举例说明

- **PRD 说**：“用户能查看历史订单，支持分页，页面响应时间 < 1s。”
- **Spec 说**：“GET /api/orders?page=1&size=20，返回订单列表及总数。page ≤ 100。若未登录返回 401。”
- **技术设计文档**：
  - 数据库索引设计（依据 Spec 的查询条件）
  - 缓存策略（依据 PRD 的 1s 响应要求）
  - 分页实现方式（依据 Spec 的 page/size 定义）

### 结论

- **主要依据：Spec**（它完整定义了要“实现什么”）
- **辅助参考：PRD**（用于理解非功能需求、优先级、业务上下文）

没有 Spec 时，技术设计只能基于模糊的 PRD，容易遗漏边界条件和异常处理。所以规范流程中，Spec 是技术设计的前提。
