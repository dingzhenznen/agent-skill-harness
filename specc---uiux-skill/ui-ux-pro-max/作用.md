• 这个 skill 的定位很明确：它是一个“大而全的 UI/UX 设计知识库 + 规则引擎”，用来指导 AI 或开发者做网页
  和移动端界面设计与实现，而不是单纯做“有审美的前端页面”。

  来源：https://skills.sh/nextlevelbuilder/ui-ux-pro-max-skill/ui-ux-pro-max
  (https://skills.sh/nextlevelbuilder/ui-ux-pro-max-skill/ui-ux-pro-max)

  它的核心作用有三层：

  1. 设计决策库
     它内置很多可检索的设计资产和规则：50+ 设计风格、161 套配色、57 组字体搭配、99 条 UX 规范、25 种图
     表类型，还覆盖 React、Next.js、Vue、Svelte、SwiftUI、React Native、Flutter、Tailwind、shadcn/ui、
     HTML/CSS 等 10 类技术栈。也就是说，它不只是告诉你“页面要高级一点”，而是试图系统化地回答：

  - 这个产品类型适合什么风格
  - 该用什么配色/排版/布局
  - 哪些交互和组件模式更合适
  - 在不同平台上要遵守哪些习惯

  2. 质量审查框架
     这个 skill 很强调“优先级规则”，把 UI/UX 决策按重要程度排序，先看无障碍、触控交互、性能，再看风
     格、布局、排版、动效、表单、导航、图表。
     这意味着它很适合拿来做：

  - 设计前的约束框架
  - 开发中的实现规范
  - 开发后的 UI/UX review checklist

  它特别重视这些硬规则：

  - 无障碍：对比度、键盘导航、aria、alt、reduced motion
  - 触控交互：44x44 点击区、8px 间距、点击反馈、避免只靠 hover
  - 性能：图片优化、CLS 控制、懒加载、骨架屏、输入延迟
  - 响应式：移动优先、无横向滚动、合理断点、可读性
  - 平台适配：iOS HIG、Material Design、React Native 等

  3. 设计系统生成器倾向
     它的描述里提到会基于产品类型和关键词推荐 patterns、colors、typography、effects，还支持 light/dark
     mode 和平台差异。这说明它不仅适合做单页，也适合做：

  - 设计系统
  - 组件库
  - 后台/仪表盘规范
  - 跨平台统一体验

  你可以把它理解成：

  - design-taste-frontend 更像“高审美前端导演”
  - ui-ux-pro-max 更像“系统化 UI/UX 设计总规范”

  - design-taste-frontend 强在审美、布局气质、前端呈现质量、反模板化
  - ui-ux-pro-max 强在知识覆盖、规范完整性、平台适配、可审查性、可复用性

  所以这个 skill 更适合这些场景：

  - 你要做的不只是一个漂亮页面，而是一整套靠谱 UI/UX
  - 你需要兼顾 Web 和 Mobile
  - 你想让 AI 有明确的无障碍、交互、响应式、性能约束
  - 你在做 dashboard、admin、SaaS、组件库、设计系统
  - 你要 review 现有页面哪里“不专业”但说不清原因

  不太适合的场景：

  - 你只是想快速做一个有视觉冲击力的营销页
  - 你更看重风格突破，而不是规范完整
  - 任务纯后端、API、数据库、脚本自动化

  如果你问“它和 spec/UI 文档工作流怎么配”，我的判断是：

  - 用它做 spec -> UI设计文档 很合适
  - 用它做 UI实现前的审查规则 也很合适
  - 单独拿它直接生成页面也可以，但生成结果可能更“规范正确”，不一定像 design-taste-frontend 那样更有强
    烈设计感

  比较好的组合方式是：

  1. 用 ui-ux-pro-max 生成或审查 UI 设计文档
  2. 再用 design-taste-frontend 提升前端落地的审美和表现力
  3. 最后再用 ui-ux-pro-max 反向 review 一遍无障碍、响应式、交互和性能

› 你的意思是 我可以使用这个skill根据spec 先生成 ui/ux设计文档,然后用这个文档指导 design-taste-
  frontend 来生成页面？


• 对，这就是比较合理的用法，而且通常比直接一步到位更稳定。

  推荐流程是：

  1. 用 ui-ux-pro-max 基于 spec 生成 UI/UX 设计文档
  2. 人工快速审一下结构、流程、状态是否合理
  3. 再把这份文档交给 design-taste-frontend 去实现页面
  4. 最后可再用 ui-ux-pro-max 做一轮 review

  这样分工很清楚：

  - spec 定义业务需求
  - ui-ux-pro-max 把需求整理成更系统的 UI/UX 方案
  - design-taste-frontend 把方案做成更有审美、更像成熟产品的前端页面

  这个组合的优势是：

  - 前一步先把信息架构、交互、状态、响应式理清
  - 后一步专注视觉表达和前端实现质量
  - 能减少 design-taste-frontend 因为“猜需求”而跑偏
  - 产出通常会同时兼顾规范性和设计感

  一句话概括：

  - ui-ux-pro-max 更像“设计规划师”
  - design-taste-frontend 更像“高审美前端实现师”

  如果你直接这样喂代理，效果通常不错：

  第一步：
  根据 spec，用 ui-ux-pro-max 的方法生成一份完整 UI/UX 设计文档。

  第二步：
  严格基于这份 UI/UX 设计文档，使用 design-taste-frontend 的设计和实现原则完成前端页面，不要偏离信息架
  构和核心交互。

  再补一个关键点：

  最好不要让第二步完全自由发挥，应该明确限制它：

  - 不改变页面目标
  - 不重构信息架构
  - 不删关键状态
  - 只在视觉层次、布局节奏、组件表现、动效细节上增强
